## 🔴 Critical 병목 지점 (우선순위 1)

### 1. **임베딩 생성 - 순차적 배치 처리**

**위치**: `src/lib/services/RAGProcessor.ts:229-291`

**문제점**:
- 배치 크기는 25개로 증가했지만, **배치 간 순차 처리**로 인해 전체 시간이 길어짐
- 각 청크당 임베딩 생성 시간: **0.5-2초**
- 청크 100개 기준: 4개 배치 × 2초 = **8초 이상** (이상적인 경우)
- 실제로는 각 청크가 순차적으로 처리되어 **50-200초 (1-3분)** 소요

**현재 코드**:
```typescript
const BATCH_SIZE = 25;
for (let i = 0; i < chunks.length; i += BATCH_SIZE) {
  const batch = chunks.slice(i, i + BATCH_SIZE);
  // 배치 내에서는 병렬 처리
  const batchPromises = batch.map(async (chunk) => {
    const result = await embeddingService.generateEmbedding(chunk.content);
    // ...
  });
  await Promise.all(batchPromises); // 배치 완료 대기
  // 다음 배치는 이전 배치가 완료된 후 시작
}
```

**영향**:
- 청크 50개: 약 25-100초
- 청크 100개: 약 50-200초
- 청크 500개: 약 250-1000초 (4-16분)

### 2. **BGE-M3 모델 초기화 지연**

**위치**: `src/lib/services/EmbeddingService.ts:24-57`

**문제점**:
- 첫 임베딩 생성 시 모델 다운로드 및 초기화 필요
- 초기화 시간: **10-30초** (네트워크 상태에 따라)
- 서버리스 환경에서는 **매 요청마다 초기화**될 수 있음 (콜드 스타트)

**현재 코드**:
```typescript
async initialize(model: string = 'bge-m3'): Promise<void> {
  // 모델 다운로드 및 로딩
  this.pipeline = await pipeline('feature-extraction', 'Xenova/bge-m3', {
    quantized: true,
    cache_dir: './.cache/transformers',
  });
}
```

**영향**:
- 첫 임베딩 생성 시: **10-30초 추가 지연**
- 콜드 스타트 시: **매번 초기화 필요**

---

### 3. **청킹 처리 복잡도**

**위치**: `src/lib/services/RAGProcessor.ts:867-1225`

**문제점**:
- 여러 단계의 청킹 로직 (UnifiedChunkingService → AdaptiveChunkingService → 강제 재청킹)
- 강제 재청킹 로직이 복잡하고 시간 소모적
- 청크 검증 및 재처리 로직이 많음

**현재 처리 흐름**:
1. UnifiedChunkingService로 초기 청킹
2. 결과 검증 (1개 청크인 경우 강제 재청킹)
3. 강제 재청킹 시도 (여러 단계)
4. 최종 검증

**영향**:
- 작은 문서 (10KB): 약 1-2초
- 중간 문서 (100KB): 약 3-5초
- 큰 문서 (1MB+): 약 10-20초

## 🟡 Major 병목 지점 (우선순위 2)

### 4. **데이터베이스 청크 저장 - 순차적 배치**

**위치**: `src/lib/services/RAGProcessor.ts:537-723`

**문제점**:
- 청크 500개 이상일 때만 배치 처리 (50개씩)
- 배치 간 **200ms 대기** 시간 추가
- 각 배치마다 DB 쿼리 실행

**현재 코드**:
```typescript
if (isLargeBatch) {
  const batchSize = 50;
  for (let i = 0; i < chunkInserts.length; i += batchSize) {
    const batch = chunkInserts.slice(i, i + batchSize);
    await supabase.from('document_chunks').insert(batch);
    // 배치 간 대기
    if (i + batchSize < chunkInserts.length) {
      await new Promise(resolve => setTimeout(resolve, 200));
    }
  }
}
```

**영향**:
- 청크 100개: 약 2-5초
- 청크 500개: 약 10-25초 (배치 대기 시간 포함)

## 📈 성능 개선 권장사항

### 우선순위 1: 임베딩 생성 최적화

1. **배치 간 병렬 처리**
   - 현재: 배치 1 → 배치 2 → 배치 3 (순차)
   - 개선: 배치 1, 2, 3 동시 처리 (병렬)
   - 예상 효과: **50-70% 시간 단축**

2. **임베딩 모델 캐싱**
   - 서버리스 환경에서 모델 재사용
   - Vercel Edge Functions 또는 별도 서버 고려
   - 예상 효과: **초기화 시간 제거 (10-30초 절약)**

3. **작은 청크는 해시 기반 임베딩**
   - 100자 이하 청크는 해시 기반으로 빠르게 처리
   - 예상 효과: **작은 청크 처리 시간 90% 단축**

---

### 우선순위 2: 청킹 처리 최적화

1. **청킹 로직 단순화**
   - 강제 재청킹 로직 최적화
   - 불필요한 검증 단계 제거
   - 예상 효과: **청킹 시간 30-50% 단축**

2. **청크 크기 동적 조정**
   - 문서 크기에 따라 청크 크기 자동 조정
   - 예상 효과: **청크 개수 최적화, 임베딩 시간 단축**

---

### 우선순위 3: 데이터베이스 최적화

1. **청크 저장 배치 크기 증가**
   - 현재: 50개씩
   - 개선: 100-200개씩 (DB 성능에 따라)
   - 예상 효과: **저장 시간 30-50% 단축**

2. **배치 간 대기 시간 제거**
   - 현재: 200ms 대기
   - 개선: 대기 없이 연속 처리
   - 예상 효과: **저장 시간 20-30% 단축**

---

### 우선순위 4: 큐 처리 최적화

1. **큐 워커 즉시 실행 보장**
   - 큐 등록 후 즉시 워커 트리거
   - 예상 효과: **큐 대기 시간 제거 (1-60초 절약)**

2. **큐 처리 상태 실시간 업데이트**
   - WebSocket 또는 Server-Sent Events 사용
   - 예상 효과: **폴링 오버헤드 제거**
