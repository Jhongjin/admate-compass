# BGE-M3 모델 캐시 문제 상세 설명

## ❓ 질문: 매번 크롤링할 때마다 BGE-M3 모델을 다운로드해야 하는 이유

한번 다운로드된 모델은 이후부터는 바로 사용할 수 있는 구조가 아닌지?

---

## 🔍 근본 원인 분석

### 1. **Vercel 서버리스 환경의 `/tmp` 디렉토리 특성**

**현재 코드:**
```typescript
// src/lib/services/EmbeddingService.ts:40
const cacheDir = isVercel ? '/tmp/.cache/transformers' : './.cache/transformers';
```

**문제점:**
- `/tmp` 디렉토리는 **서버리스 함수의 임시 저장소**
- **요청 간에 유지되지 않을 수 있음** (Vercel의 서버리스 함수는 요청 종료 후 정리)
- 각 요청마다 모델을 다시 다운로드해야 할 수 있음
- **캐시 히트율이 매우 낮음** (거의 0%에 가까움)

### 2. **서버리스 함수의 생명주기**

```
요청 1: 함수 시작 → 모델 다운로드 (/tmp/.cache/transformers) → 처리 완료 → 함수 종료 → /tmp 정리
요청 2: 함수 시작 → /tmp가 비어있음 → 모델 다시 다운로드 → 처리 완료 → 함수 종료
```

**특징:**
- **Cold Start**: 새로운 요청 시 모듈이 다시 로드되면서 싱글톤 인스턴스가 재생성됨
- **Warm Start**: 같은 인스턴스가 재사용되면 싱글톤이 유지되지만, **보장되지 않음**
- **함수 종료 시**: `/tmp` 디렉토리가 정리될 수 있음

### 3. **캐시 디렉토리 지속성 문제**

**이론적으로는:**
- 한번 다운로드된 모델은 `/tmp/.cache/transformers`에 저장됨
- 다음 요청에서 같은 인스턴스가 재사용되면 캐시에서 로드 가능

**실제로는:**
- Vercel 서버리스 함수는 **요청 간에 상태를 보장하지 않음**
- `/tmp` 디렉토리는 **요청 종료 후 정리될 수 있음**
- **다른 인스턴스**가 요청을 처리하면 캐시가 없음
- **Cold Start** 시 항상 모델을 다시 다운로드해야 함

## 📊 실제 동작 시나리오

### 시나리오 1: 같은 인스턴스 재사용 (Warm Start)
```
요청 1: 모델 다운로드 (30-70초) → /tmp에 저장 → 처리 완료
요청 2: 같은 인스턴스 → /tmp에서 모델 로드 (10-20초) → 처리 완료 ✅
```

**확률**: 약 20-30% (Vercel이 인스턴스를 재사용하는 경우)

### 시나리오 2: 다른 인스턴스 사용 (Cold Start)
```
요청 1: 모델 다운로드 (30-70초) → /tmp에 저장 → 처리 완료 → 함수 종료
요청 2: 새로운 인스턴스 → /tmp가 비어있음 → 모델 다시 다운로드 (30-70초) ❌
```

**확률**: 약 70-80% (대부분의 경우)

### 시나리오 3: /tmp 정리됨
```
요청 1: 모델 다운로드 → /tmp에 저장 → 처리 완료 → 함수 종료 → /tmp 정리
요청 2: 같은 인스턴스라도 → /tmp가 비어있음 → 모델 다시 다운로드 ❌
```

**확률**: 불확실 (Vercel의 정리 정책에 따라 다름)

## 🔧 해결 방안 비교

### 옵션 1: OpenAI Embeddings API 사용 (현재 적용됨) ✅

**장점:**
- ✅ **즉시 사용 가능** (초기화 불필요)
- ✅ **안정적** (네트워크 문제 없음)
- ✅ **빠름** (API 호출만으로 처리)
- ✅ **정확도 유지** (검증된 모델)
- ✅ **서버리스 환경에 최적화**

**단점:**
- ❌ 비용 발생 (월 $0.0001/1K tokens, 약 $0.10/1M tokens)
- ❌ 외부 API 의존성

**비용 예시:**
- 문서 1개 (13개 청크): 약 $0.0000013 (거의 무료)
- 문서 100개 (1,300개 청크): 약 $0.00013
- 문서 10,000개 (130,000개 청크): 약 $0.013

### 옵션 2: 외부 스토리지에 모델 저장 (Supabase Storage, S3 등)

**구현 방법:**
```typescript
// 1. 모델을 Supabase Storage에 업로드
// 2. 요청 시 Storage에서 다운로드 (캐시 확인)
// 3. /tmp에 복사 후 사용
```

**장점:**
- ✅ 영구 저장 (요청 간 유지)
- ✅ 여러 인스턴스에서 공유 가능

**단점:**
- ❌ 구현 복잡도 높음
- ❌ Storage에서 다운로드 시간 (여전히 30-70초)
- ❌ Storage 비용 발생
- ❌ 여전히 느림

### 옵션 3: 별도 서버에서 모델 호스팅

**구현 방법:**
- FastAPI 서버에서 모델을 항상 메모리에 로드
- API로 임베딩 요청

**장점:**
- ✅ 빠른 응답 (모델이 항상 메모리에 있음)
- ✅ 캐시 효율적

**단점:**
- ❌ 인프라 관리 필요
- ❌ 비용 증가 (서버 유지비)
- ❌ 구현 복잡도 높음

## 💡 결론 및 권장사항

### 현재 상황
- **BGE-M3 모델은 서버리스 환경에서 캐시가 제대로 작동하지 않음**
- `/tmp` 디렉토리는 요청 간에 유지되지 않을 수 있음
- **매번 다운로드해야 하는 이유**: Vercel 서버리스 환경의 제약

### 권장사항
**OpenAI Embeddings API를 기본값으로 사용** (현재 적용됨)

**이유:**
1. ✅ **즉시 사용 가능**: 초기화 시간 0초
2. ✅ **안정적**: 네트워크 문제로 멈추지 않음
3. ✅ **빠름**: API 호출만으로 처리 (1-2초)
4. ✅ **정확도 유지**: 검증된 모델 사용
5. ✅ **비용 합리적**: 월 $0.10/1M tokens (거의 무료)

### BGE-M3를 사용해야 하는 경우
- **로컬 개발 환경**: `./.cache/transformers` 사용 (영구 저장)
- **전용 서버**: 모델을 항상 메모리에 로드
- **Edge Functions**: Vercel Edge Functions 사용 (제한적)
