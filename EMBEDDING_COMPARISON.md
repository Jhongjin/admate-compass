# 임베딩 방식 비교: 해시 기반 vs BGE-M3

## 📊 개요

현재 시스템은 두 가지 임베딩 방식을 사용합니다:
1. **BGE-M3**: 딥러닝 기반 의미 임베딩 (기본)
2. **해시 기반 임베딩**: Fallback용 빠른 임베딩

---

## 🔵 BGE-M3 임베딩

### 작동 원리
- **딥러닝 모델**: BAAI General Embedding Model 3 (BGE-M3)
- **차원**: 1024차원 벡터
- **의미 이해**: 텍스트의 의미적 유사도를 학습하여 벡터로 표현
- **모델 크기**: 약 500MB (quantized 버전)

### 장점 ✅
1. **높은 의미 검색 품질**
   - 동의어, 유사어, 문맥을 이해
   - 예: "광고 집행"과 "광고 운영"을 유사한 의미로 인식
   - 예: "비용 절감"과 "예산 절약"을 유사한 의미로 인식

2. **정확한 검색 결과**
   - 사용자 질문과 문서 내용의 의미적 유사도 측정
   - 관련성 높은 문서를 우선적으로 반환

3. **다국어 지원**
   - 한국어, 영어 등 다양한 언어 지원
   - 언어 간 의미적 유사도 측정 가능

### 단점 ❌
1. **초기화 시간이 오래 걸림**
   - Cold Start: 40-90초 (모델 다운로드 + 로딩)
   - Warm Start: 10-20초 (캐시된 모델 로딩)
   - Vercel 서버리스 환경에서는 매 요청마다 초기화될 수 있음

2. **메모리 사용량**
   - 약 200-300MB 메모리 사용
   - Vercel 서버리스 함수 메모리 제한 (1024MB) 고려 필요

3. **처리 시간**
   - 각 청크당 0.5-2초 소요
   - 청크 100개 기준: 약 50-200초 (1-3분)

4. **네트워크 의존성**
   - 모델 다운로드 시 네트워크 상태에 따라 시간 변동
   - 불안정한 네트워크 환경에서 실패 가능

## ⚡ 해시 기반 임베딩

### 작동 원리
```typescript
// 간단한 해시 함수로 텍스트를 숫자로 변환
const hash = simpleHash(text); // 예: "광고 집행" → 1234567

// 해시값을 기반으로 1024차원 벡터 생성 (의사 랜덤)
for (let i = 0; i < 1024; i++) {
  const seed = (hash + i * 17) % 1000000;
  embedding[i] = (Math.sin(seed) * 0.5 + 0.5) * 2 - 1; // -1 ~ 1 범위
}
```

- **해시 함수**: 텍스트를 고유한 숫자로 변환
- **벡터 생성**: 해시값을 기반으로 수학적 함수로 1024차원 벡터 생성
- **의미 이해 없음**: 단순히 텍스트의 문자 패턴만 반영

### 장점 ✅
1. **매우 빠른 처리**
   - 초기화 시간: 0초 (즉시 사용 가능)
   - 각 청크당: < 1ms (거의 즉시)
   - 청크 100개 기준: 약 0.1초

2. **안정성**
   - 네트워크 의존성 없음
   - 외부 서비스 불필요
   - 항상 사용 가능

3. **비용 없음**
   - 추가 인프라 불필요
   - API 호출 비용 없음

4. **메모리 효율**
   - 메모리 사용량: 거의 0MB
   - 서버리스 환경에 최적화

### 단점 ❌
1. **의미 검색 품질이 낮음**
   - 동의어, 유사어를 인식하지 못함
   - 예: "광고 집행"과 "광고 운영"을 다른 의미로 인식
   - 예: "비용 절감"과 "예산 절약"을 다른 의미로 인식

2. **검색 정확도 저하**
   - 문자 매칭에 의존 (정확한 단어 일치 필요)
   - 관련성 높은 문서를 놓칠 수 있음
   - 사용자 질문과 문서 내용의 의미적 유사도 측정 불가

3. **동일 텍스트만 매칭**
   - 완전히 동일한 텍스트만 유사도 높음
   - 약간 다른 표현은 다른 벡터로 인식

## 🎯 사용 시나리오

### BGE-M3 사용 권장
- ✅ **프로덕션 환경** (안정적인 서버)
- ✅ **의미 검색 품질이 중요한 경우**
- ✅ **충분한 초기화 시간이 있는 경우**
- ✅ **사용자 질문의 정확한 답변 필요**

### 해시 기반 임베딩 사용 권장
- ✅ **Fallback 상황** (BGE-M3 초기화 실패)
- ✅ **빠른 문서 인덱싱이 필요한 경우**
- ✅ **서버리스 환경** (타임아웃 제한)
- ✅ **임시/테스트 환경**

## 🔄 현재 시스템의 동작 방식

### 1. 우선순위: BGE-M3
```
시작 → BGE-M3 초기화 시도 (1분 30초 타임아웃)
  ↓
성공 → BGE-M3로 임베딩 생성 (높은 품질)
  ↓
실패/타임아웃 → 해시 기반 임베딩으로 자동 전환 (빠른 처리)
```

### 2. 하이브리드 방식
- **작은 청크 (100자 이하)**: 항상 해시 기반 임베딩 사용 (빠른 처리)
- **큰 청크 (100자 초과)**: BGE-M3 사용 (의미 검색 품질)

### 3. Fallback 메커니즘
- BGE-M3 초기화 실패 시 자동으로 해시 기반 임베딩 사용
- 문서 인덱싱은 완료되지만 검색 품질은 낮아짐

## 💡 권장 사항

### 단기 (현재)
1. **해시 기반 임베딩으로 문서 인덱싱 완료** ✅
   - 크롤링이 빠르게 진행되어 서브 페이지까지 처리 가능
   - 문서가 인덱싱되어 기본 검색 가능

2. **BGE-M3 초기화는 백그라운드에서 계속 시도**
   - 초기화 완료 후 재인덱싱 고려

### 중기 (개선)
1. **BGE-M3 모델 프리로딩**
   - 별도 서버에서 모델을 항상 메모리에 로드
   - API 서버로 모델 제공

2. **외부 임베딩 서비스 사용**
   - OpenAI Embeddings API
   - Supabase Vector
   - 안정적이고 빠른 처리

### 장기 (최적화)
1. **하이브리드 검색**
   - 키워드 검색 (해시 기반) + 의미 검색 (BGE-M3)
   - 두 결과를 결합하여 검색 품질 향상

2. **캐싱 전략**
   - 자주 사용되는 질문의 임베딩 캐싱
   - 검색 속도 향상
