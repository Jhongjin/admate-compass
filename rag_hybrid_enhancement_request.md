# RAG 시스템 고도화 개발 요청서
## 벡터DB + RAG + 멀티LLM Ops 기반 하이브리드 검색 구조

---

## 1. 개요

### 1.1 현황 및 문제점

현재 RAG 시스템은 키워드 매칭 중심의 검색 방식으로 운영되고 있어, 사용자의 실제 의도와 다른 단순한 결과가 반환되는 문제가 발생하고 있습니다.

**현재 구조의 한계:**

| 문제 유형 | 사례 | 원인 |
|----------|------|------|
| 키워드 불일치 | "배너 사이즈 알려줘" → 사이즈 관련 없는 배너 상품 정보 반환 | 키워드 매칭만으로는 "사이즈"의 맥락 파악 불가 |
| 유사 의미 미인식 | "단가" 검색 시 "가격", "비용", "CPM" 등 유사 표현 누락 | 동의어/유사어 의미 연결 부재 |
| 복합 질문 실패 | "네이버와 카카오 DA 단가 비교" → 한쪽 정보만 반환 | 다중 엔티티 질문 처리 미지원 |
| 맥락 단절 | 이전 대화 내용을 이어가는 질문에 엉뚱한 답변 | 대화 히스토리 컨텍스트 미반영 |
| 할루시네이션 | 문서에 없는 정보를 사실처럼 답변 | 단일 LLM 의존, 검증 체계 부재 |

### 1.2 고도화 목적

키워드 중심 검색에서 **의미(Semantic) 기반 검색**으로 전환하고, **벡터DB + RAG + 멀티LLM Ops** 하이브리드 구조를 통해 사용자 의도에 맞는 신뢰성 있는 답변을 생성하는 시스템으로 고도화합니다.

### 1.3 기대 효과

| 지표 | 현재 (AS-IS) | 목표 (TO-BE) |
|------|-------------|-------------|
| 답변 정확도 | 약 60~70% | 90% 이상 |
| 의도 파악 성공률 | 약 50% | 85% 이상 |
| 할루시네이션 발생률 | 약 15~20% | 5% 이하 |
| 복합 질문 처리율 | 미지원 | 80% 이상 |
| 평균 응답 만족도 | 낮음 | 높음 |

## 2. 고도화 아키텍처

### 2.1 전체 구조: 하이브리드 검색 + 멀티LLM Ops

```
┌─────────────────────────────────────────────────────────────────────┐
│                        사용자 질문 입력                               │
└──────────────────────────────┬──────────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    ① Query Understanding Layer                      │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────────────────┐ │
│  │ 의도 분류     │  │ 키워드 추출   │  │ Query Rewriting/Expansion │ │
│  └──────────────┘  └──────────────┘  └───────────────────────────┘ │
└──────────────────────────────┬──────────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                ② Hybrid Retrieval Layer (병렬 검색)                  │
│  ┌────────────────────┐          ┌────────────────────┐            │
│  │  Semantic Search    │          │  Keyword Search     │            │
│  │  (벡터 유사도 검색)  │          │  (BM25 역인덱스)     │            │
│  │  - 의미 기반 매칭    │          │  - 정확 키워드 매칭   │            │
│  │  - 벡터DB (Top-K)   │          │  - Elasticsearch    │            │
│  └─────────┬──────────┘          └──────────┬─────────┘            │
│            └──────────┬─────────────────────┘                      │
│                       ▼                                             │
│             ┌──────────────────┐                                    │
│             │  RRF 스코어 병합   │  (Reciprocal Rank Fusion)         │
│             └──────────────────┘                                    │
└──────────────────────────────┬──────────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    ③ Re-Ranking Layer                                │
│  ┌──────────────────────────────────────────────────────────┐      │
│  │  Cross-Encoder 기반 정밀 재순위 (Cohere Rerank 등)         │      │
│  │  → 질문-문서 쌍별 관련도 정밀 평가 후 최종 Top-K 선정       │      │
│  └──────────────────────────────────────────────────────────┘      │
└──────────────────────────────┬──────────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                  ④ Multi-LLM Ops Layer                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────────┐ │
│  │ 답변 생성 LLM │  │ 검증 LLM     │  │ 라우팅 / 폴백 관리        │ │
│  │ (Primary)     │  │ (Validator)  │  │ (Router + Fallback)      │ │
│  └──────────────┘  └──────────────┘  └──────────────────────────┘ │
└──────────────────────────────┬──────────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                     ⑤ 최종 답변 + 출처 제공                          │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 각 레이어별 상세 설명

#### ① Query Understanding Layer — 질문 분석 및 변환

사용자 질문을 LLM이 사전 분석하여, 검색 품질을 높이기 위한 전처리를 수행합니다.

| 기능 | 설명 | 예시 |
|------|------|------|
| 의도 분류 | 질문의 목적을 분류 (단가 조회, 비교, 가이드 확인 등) | "배너 사이즈" → 제작가이드 조회 의도 |
| 키워드 추출 | 핵심 엔티티(상품명, 미디어명, 속성) 추출 | "네이버 브랜드검색 수수료" → [네이버, 브랜드검색, 수수료] |
| Query Rewriting | 질문을 검색에 최적화된 형태로 변환 | "얼마예요?" → 구체적 단가/수수료 조회 쿼리로 변환 |
| Query Expansion | 동의어/유사어로 검색 범위 확장 | "단가" → "단가, 가격, CPM, CPV, 비용" |
| 멀티턴 컨텍스트 | 이전 대화 맥락을 반영한 질문 보완 | "그럼 카카오는?" → "카카오 DA 단가는?" |

#### ② Hybrid Retrieval Layer — 하이브리드 검색

의미 검색과 키워드 검색을 병렬로 수행한 후, 결과를 통합합니다.

**Semantic Search (벡터 유사도 검색):**
- 질문과 문서 청크를 동일한 임베딩 공간에서 비교
- 동의어, 유사 표현, 맥락적으로 관련된 내용까지 검색 가능
- 벡터DB에서 코사인 유사도 기반 Top-K 검색

**Keyword Search (BM25 기반):**
- 정확한 키워드 매칭이 중요한 경우 (상품명, 고유명사, 숫자 등)
- Elasticsearch 또는 동등한 역인덱스 검색 엔진 활용
- 정확한 용어가 포함된 문서를 놓치지 않음

**RRF (Reciprocal Rank Fusion) 스코어 병합:**
- 두 검색 결과의 순위를 통합하여 최종 후보 선정
- 의미적으로 관련되면서도 키워드가 정확히 매칭되는 결과를 상위로 배치

#### ③ Re-Ranking Layer — 정밀 재순위

| 항목 | 설명 |
|------|------|
| 방식 | Cross-Encoder 모델이 질문-문서 쌍을 직접 비교하여 관련도 점수 산출 |
| 목적 | 초기 검색의 노이즈를 제거하고, 가장 관련성 높은 청크만 LLM에 전달 |
| 권장 도구 | Cohere Rerank, BGE-Reranker, 또는 자체 Fine-tuned Cross-Encoder |
| 효과 | 검색 정밀도(Precision) 대폭 향상, LLM 입력 토큰 최적화 |

#### ④ Multi-LLM Ops Layer — 멀티LLM 운영 체계

단일 LLM 의존에서 벗어나, 역할별 LLM을 분리하고 운영 안정성을 확보합니다.

**LLM 역할 분리:**

| 역할 | 설명 | 권장 모델 |
|------|------|----------|
| Primary (답변 생성) | 검색된 컨텍스트 기반 최종 답변 생성 | Claude Sonnet / GPT-4o |
| Validator (검증) | 생성된 답변이 원문 근거와 일치하는지 검증 | Claude Haiku / GPT-4o-mini (경량 모델) |
| Query Analyzer (질문 분석) | Query Understanding 단계 처리 | Claude Haiku / GPT-4o-mini (경량 모델) |

**운영 안정성 기능:**

| 기능 | 설명 |
|------|------|
| LLM 라우팅 | 질문 복잡도에 따라 적합한 모델로 자동 라우팅 (단순 질문 → 경량 모델, 복합 질문 → 고성능 모델) |
| 폴백 처리 | Primary LLM 장애 시 자동으로 Secondary LLM으로 전환 |
| 속도 제어 | Rate Limiting, 큐잉을 통한 API 호출 안정적 관리 |
| 비용 최적화 | 질문 유형별 모델 분배로 불필요한 고비용 모델 호출 최소화 |

## 3. 데이터 벡터화 고도화

### 3.1 문서 전처리 개선

현재 단순 텍스트 추출 방식에서, 구조화된 의미 단위 추출로 전환합니다.

| 구분 | 현재 (AS-IS) | 고도화 (TO-BE) |
|------|-------------|---------------|
| PDF 파싱 | 단순 텍스트 추출 | 테이블/이미지 OCR 포함 구조화 파싱 (Unstructured, LlamaParse 등) |
| Excel 파싱 | 시트 단위 텍스트 변환 | 시트명+헤더+행 단위 의미 보존 파싱, 병합셀 처리 |
| 청킹 방식 | 고정 사이즈 분할 | 의미 단위 청킹 (Semantic Chunking) + 테이블 전용 청킹 |
| 메타데이터 | 파일명 정도만 저장 | 문서명, 페이지, 섹션, 카테고리, 상품명, 미디어명 등 풍부한 메타데이터 |

### 3.2 임베딩 전략 개선

| 항목 | 상세 |
|------|------|
| 임베딩 모델 | 한국어 성능 우수 모델 선택 (OpenAI text-embedding-3-large, Cohere embed-v3 등) |
| 멀티벡터 | 동일 청크에 대해 원문 벡터 + 요약 벡터를 함께 저장하여 검색 다양성 확보 |
| 메타데이터 필터 | 벡터 검색 시 카테고리, 문서유형, 미디어명 등으로 사전 필터링 가능 |
| 주기적 갱신 | 문서 업데이트 시 변경된 청크만 증분 임베딩 (전체 재생성 방지) |

### 3.3 의미 단위 연결 (Knowledge Graph 연계)

단순 청크 나열이 아닌, 청크 간 의미적 관계를 구조화합니다.

- **엔티티 연결**: 상품명 ↔ 단가 ↔ 미디어 ↔ 수수료 등 관계 매핑
- **상위/하위 청크 연결**: Parent-Child 관계를 통해 세부 정보와 전체 맥락을 동시 참조
- **크로스 문서 연결**: 서로 다른 문서에 분산된 동일 상품/미디어 정보를 연결

## 4. 답변 신뢰성 강화

### 4.1 할루시네이션 방지 체계

| 단계 | 방법 | 설명 |
|------|------|------|
| 프롬프트 제어 | Grounded Generation | 시스템 프롬프트에 "제공된 문서 내용만으로 답변" 명시, 문서 외 정보 생성 금지 |
| 답변 검증 | Validator LLM | 별도 LLM이 답변과 원문 청크를 비교하여 근거 존재 여부 검증 |
| 신뢰도 표시 | Confidence Score | 답변의 근거 강도에 따라 신뢰도 점수 (높음/보통/낮음) 표시 |
| 불확실성 처리 | Fallback 응답 | 근거 불충분 시 "해당 정보를 문서에서 찾을 수 없습니다" 안내 + 관련 문서 제안 |

### 4.2 답변 품질 평가 체계

| 평가 항목 | 측정 방식 |
|----------|----------|
| Faithfulness | 답변 내용이 원문 청크에 근거하는지 자동 평가 |
| Relevance | 답변이 질문 의도에 부합하는지 자동 평가 |
| Completeness | 답변이 질문에 필요한 정보를 충분히 포함하는지 평가 |
| 사용자 피드백 | 👍/👎 피드백 수집 → 정확도 개선 데이터로 활용 |

## 5. 예상 사용 시나리오 (고도화 후)

### 시나리오 1: 의미 기반 검색

> **질문**: "카카오 모먼트에서 영상 광고 돌리려면 비용이 얼마나 들어요?"
>
> **현재**: "카카오 모먼트" 키워드로만 검색 → 관련 없는 카카오 모먼트 일반 정보 반환
>
> **고도화 후**: 의도(단가 조회) 파악 → "카카오모먼트 + 영상 + CPV/단가" 의미 검색 → 정확한 영상 광고 단가표 반환 + 출처 페이지 표시

### 시나리오 2: 복합 비교 질문

> **질문**: "네이버 브랜드검색이랑 카카오 브랜드검색 수수료 비교해줘"
>
> **현재**: 한쪽 정보만 부분 반환 또는 실패
>
> **고도화 후**: 두 미디어의 수수료 정보를 각각 검색 → 표 형태로 비교 답변 생성 + 각 출처 표시

### 시나리오 3: 멀티턴 대화

> **질문 1**: "XX 상품 단가 알려줘"
> **답변 1**: "XX 상품의 CPM 단가는 ₩5,000입니다. (출처: 2026년 상반기 단가표 p.12)"
>
> **질문 2**: "거기서 수수료는?"
>
> **현재**: "거기서"의 맥락을 이해하지 못해 실패
>
> **고도화 후**: 이전 대화에서 "XX 상품"을 참조 → "XX 상품 수수료율은 15%입니다" 정확 응답

### 시나리오 4: 할루시네이션 방지

> **질문**: "2026년 하반기 네이버 단가 인상 예정이야?"
>
> **현재**: 문서에 없는 정보를 추측하여 답변 (할루시네이션)
>
> **고도화 후**: Validator가 근거 부재 확인 → "등록된 문서에서 2026년 하반기 단가 인상 관련 정보를 찾을 수 없습니다. 현재 등록된 최신 단가표 기준 정보를 안내해드릴까요?"

## 6. 기술 스택 권장안

| 구성 요소 | 권장 기술 | 비고 |
|----------|----------|------|
| 벡터DB | Pinecone / Qdrant / Weaviate | 하이브리드 검색(벡터+키워드) 네이티브 지원 여부 확인 |
| 키워드 검색 | Elasticsearch / OpenSearch | BM25 검색용, 벡터DB 자체 키워드 검색 지원 시 통합 가능 |
| 임베딩 | OpenAI text-embedding-3-large / Cohere embed-v3 | 한국어 성능 벤치마크 비교 후 선정 |
| Re-Ranker | Cohere Rerank / BGE-Reranker-v2 | Cross-Encoder 기반 재순위 |
| Primary LLM | Claude Sonnet 4.5 / GPT-4o | 답변 생성 메인 모델 |
| Validator LLM | Claude Haiku / GPT-4o-mini | 답변 검증 및 질문 분석 (경량/저비용) |
| 오케스트레이션 | LangChain / LlamaIndex | RAG 파이프라인 관리 |
| 모니터링 | LangSmith / Langfuse | LLM 호출 추적, 비용 모니터링, 품질 평가 |
| 백엔드 | Python FastAPI | 기존 아키텍처 유지 |

## 7. 일정 및 마일스톤 (안)

## 8. 비용 영향 분석

### 8.1 추가 비용 항목

| 항목 | 설명 | 비용 영향 |
|------|------|----------|
| Re-Ranker API | Cohere Rerank 등 외부 서비스 | 검색 건당 소액 추가 |
| Validator LLM | 답변 검증용 경량 LLM 추가 호출 | 건당 Primary 대비 약 10~20% 추가 |
| Query Analyzer LLM | 질문 분석용 경량 LLM 호출 | 건당 소액 추가 |
| Elasticsearch | 키워드 검색 엔진 인프라 | 벡터DB 자체 지원 시 불필요 |
| 모니터링 도구 | LangSmith/Langfuse 등 | 무료 티어 또는 월정액 |

### 8.2 비용 절감 효과

| 항목 | 설명 |
|------|------|
| LLM 라우팅 | 단순 질문은 경량 모델로 처리 → 고비용 모델 호출 30~50% 절감 예상 |
| Re-Ranking | LLM에 전달하는 청크 수 최적화 → 입력 토큰 절감 |
| 캐싱 | 동일/유사 질문 캐싱으로 반복 호출 방지 |

## 9. 협의 요청 사항

### 9.1 개발사 확인 필요 사항

- 현재 벡터DB가 하이브리드 검색(벡터+키워드)을 네이티브 지원하는지 확인, 미지원 시 별도 Elasticsearch 도입 필요 여부
- Re-Ranker 모델 선정 시 한국어 성능 벤치마크 결과 공유 요청
- 멀티LLM 구성 시 API Key 관리 및 비용 분리 방안
- 기존 임베딩 데이터의 마이그레이션 전략 (전체 재생성 vs 병행 운영)
- Query Understanding 레이어의 프롬프트 최적화 협업 방안

### 9.2 운영부서 제공 예정 사항

- 기존 시스템의 오답 사례 수집 및 제공 (정확도 개선 참고)
- 고도화 후 QA 테스트용 질문-정답 셋 확장 작성 (복합질문, 비교질문 포함)
- UAT 참여 및 AS-IS/TO-BE 비교 평가 피드백
